# 法律知识库审计与丰容技能

扫描Obsidian法律知识库，识别知识缺口，按执业领域优先级批量创建经MCP工具核实的知识卡片。

## 适用场景

- 定期知识库健康检查（建议每月1次）
- 新执业领域拓展前的知识储备
- 案件代理前的相关领域知识补充
- 知识库结构重组

## 前置条件

- Obsidian法律知识库已建立（路径见项目上下文）
- MCP法律检索工具可用（元典智库+北大法宝）
- 导航中心（00-导航中心.md）已创建

## 执行步骤

### 步骤一：扫描知识库现状

```python
# 扫描各模块文件数量
import os

VAULT = "/Users/yuzhenghong/Library/Mobile Documents/iCloud~md~obsidian/Documents/法律知识库"
modules = {
    "01-案件笔记": "案件",
    "02-法律知识": "法条",
    "03-文书模板": "模板",
    "04-实务经验": "经验",
    "08-裁判规则库": "规则",
}

for folder, label in modules.items():
    path = os.path.join(VAULT, folder)
    if os.path.exists(path):
        count = sum(1 for f in os.listdir(path) if f.endswith('.md'))
        print(f"{label}: {count}篇")
```

### 步骤三：核实法律内容（必须在创建卡片之前！）

**铁律：先核实，后写入。绝不编造。**

在批量创建知识卡片之前，先核实所有待写入的法律内容：

1. **法条核实**：用 `mcp_pkulaw_get_article` 获取法条原文
2. **案例核实**：用 `mcp_pkulaw_case_search_search_case` 检索真实案例（必须有案号）
3. **司法解释核实**：用 `mcp_yuandian_search_fagui` 确认存在且现行有效
4. **找不到的内容**：标注"⚠️待核实"，不要编造

核实完成后，才进入步骤四创建知识卡片。

### 步骤四：批量创建知识卡片

使用 `delegate_task` 并行创建（最多3个子代理）：

```
delegate_task(tasks=[
    {"goal": "用MCP工具核实法条后创建3个知识卡片...", "toolsets": ["terminal", "file"]},
    {"goal": "用MCP工具核实法条后创建3个知识卡片...", "toolsets": ["terminal", "file"]},
    {"goal": "创建4个实务经验笔记...", "toolsets": ["terminal", "file"]}
])
```

**知识卡片标准格式**：
```markdown
---
tags: [标签1, 标签2]
created: YYYY-MM-DD
source: 元典智库/北大法宝
时效性: 现行有效
---

# 标题

> **法律全称**：XXX
> **施行日期**：YYYY年M月D日
> **来源**：元典智库核实

---

## 一、条文要点

### 第X条 — 标题

> 法条原文

**实务要点**：
| 要点 | 说明 |
|------|------|
| ... | ... |

---

## 二、关联法条索引

| 法条 | 主题 | 实务用途 |
|------|------|----------|
| ... | ... | ... |

---

#标签1 #标签2
```

2. 使用 `mcp_yuandian_search_fatiao` 逐条验证
3. 输出核实结果表格

详见 `legal-research` 技能的核实流程。

## Pitfalls

### 1. delegate_task超时
**问题**：3个子代理并行时，如果MCP工具响应慢，可能超时（600秒限制）
**解决**：每个子代理的任务不要超过3个知识卡片，MCP调用次数控制在10次以内

### 2. MCP工具对民法典单条检索不稳定
**问题**：`mcp_yuandian_search_fatiao` 搜索"民法典 第XXX条"有时返回410（无结果）
**解决**：改用 `mcp_yuandian_search_fagui(keyword="民法典")` 获取全文，然后从全文中提取条文。或通过司法解释、法院文件中的引用来交叉核实。

### 3. 导航中心更新冲突
**问题**：并行创建的知识卡片可能同时尝试更新导航中心
**解决**：所有卡片创建完成后再统一更新导航中心，不要在子代理中更新

### 4. 知识卡片内容绝对不能编造（最高优先级！2026-06-09血泪教训）
**问题**：凭记忆编造案例、法条、司法解释内容
**正确做法**：
1. **先核实，后写入**：创建知识卡片前，先用MCP工具核实所有法条引用
2. 找不到的法条标注"⚠️待核实"
3. 案例必须有真实案号，不能编造"张某诉某医院"等虚构案例
4. 司法解释必须确认存在且现行有效

**2026-06-09教训**：创建医疗纠纷知识卡片时，编造了指导案例内容和虚构案例，用户质疑"核实真实性了？"后不得不承认错误。

### 5. MCP工具可能找不到某些法条
**问题**：MCP工具可能找不到某些法条
**解决**：找不到的法条标注"⚠️待核实"，不要凭记忆填写。参考 `legal-analysis-pitfalls` 技能的教训。

### 5. 重复创建已有文件
**问题**：可能创建已存在的知识卡片
**解决**：创建前先用 `search_files` 检查目标路径是否已存在同名文件

### 6. 多轮丰容模式
**问题**：用户说"继续丰容"时，不知道下一批该丰容什么
**解决**：按以下顺序分轮丰容：
- **第一轮**：P0核心法律体系（公司法、知识产权、民诉、刑辩）
- **第二轮**：P1高频案由（合同、侵权、劳动、物权、担保）+ 执行/破产
- **第三轮**：P2补充领域（行政、婚姻家事、刑法、建设工程）+ 实务经验
- **后续轮次**：文书模板扩充 → 裁判规则 → 咨询储备专题

每轮完成后更新导航中心统计，让用户看到进展。用户说"继续"时自动进入下一轮。

### 7. iCloud同步确认
**问题**：创建文件后不确定是否已同步到iCloud
**解决**：用 `brctl status` 检查CloudDocs同步状态，确认 `caught-up` 和 `last-sync` 时间。Obsidian在iCloud路径下的文件创建后通常几秒内自动同步。

### 8. 子代理声称完成但字段缺失（2026-06-08教训）
**问题**：子代理报告"已添加evidence/law/tips字段"，但实际patch失败或字段未写入文件
**解决**：创建后必须用search_files验证：
```python
search_files(pattern='evidence:', path='目标文件')  # 确认字段存在
search_files(pattern='name:', path='目标文件')  # 确认案由数量
wc -l 目标文件  # 确认行数增长
```
不要相信子代理的自我报告，必须验证文件实际内容。

### 9. 批量法条核实模式
**问题**：67个案由×3-4条法条=200+条引用，逐条手动核实效率低
**解决**：用delegate_task批量核实：
```
delegate_task(goal="核实以下37条法律依据引用的正确性。使用mcp_yuandian_search_fatiao逐条验证...", toolsets=["terminal","file"])
```
核实结果输出为表格：✅正确/❌错误/⚠️未找到。错误的立即修正并推送。

### 11. README.md更新模式（2026-06-09教训）

当在涉外法律知识库（`11-涉外法律/`）中新增知识卡片时，必须同步更新README.md：

**需要更新的4个位置**：
1. **文件数量**：`文件数量：X个` → `文件数量：X+1个`
2. **文件结构**：在对应目录下添加新文件条目
3. **文件清单**：在对应章节的表格中添加新行
4. **相关链接**：添加 `[[新文件名]]` wikilink

**更新方法**：用 `sed` 命令批量替换，避免手动编辑出错

**⚠️ 注意**：中文路径和文件名在sed中需要正确处理编码

### 10. 网站丰容与知识库丰容的区别
**问题**：混淆Obsidian知识库丰容和网站内容丰容
**解决**：
- **知识库丰容**：写入Obsidian .md文件，更新导航中心，同步iCloud
- **网站丰容**：修改HTML文件中的JavaScript数据数组，推送到GitHub Pages
- 两者都需要MCP核实法条，但文件格式和部署方式完全不同
