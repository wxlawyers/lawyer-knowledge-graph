# 法律分析常见错误与教训

本技能记录法律分析中极易犯的错误，每次分析案件前必须加载并遵守。

---

## 一、举证责任分配错误（致命错误）

### 1.1 举证责任分配与鉴定申请策略

**核心原则**：根据《民事诉讼法》第67条"谁主张谁举证"原则，但**绝不能一杆子打死**，必须视案件案由、性质、争议标的而定。

**⚠️ 血泪教训（2026-06-04）**：最初写成"被告绝不能主动申请鉴定"，被律师纠正。法律规则有原则就有例外，必须具体问题具体分析。

#### 一般规则（买卖合同质量纠纷）

- 原告主张设备有质量缺陷 → 举证责任在原告 → 一般由原告申请鉴定
- 被告不需要自证清白 → 一般不主动申请鉴定

#### 被告应主动申请鉴定的情形

1. **被告主张过错相抵**：被告需申请鉴定证明原告过错（如安装不当、超工况运行）——此时举证责任在被告
2. **被告主张设备符合标准**：主动申请鉴定可选择有利鉴定机构
3. **预判原告会申请**：不如被告先申请，掌握主动权
4. **争议标的较大**：鉴定结论对案件走向影响大时，值得主动申请
5. **案由本身要求被告举证**：如产品责任侵权等举证责任倒置情形

#### 举证责任倒置的情形（法定例外，被告必须举证）

| 案由 | 倒置内容 | 法律依据 |
|------|---------|---------|
| 产品责任侵权 | 生产者证明无缺陷 | 产品质量法第41条 |
| 环境污染侵权 | 污染者证明无因果关系 | 民法典第1230条 |
| 高度危险作业 | 占有者/使用者证明已尽注意义务 | 民法典第1236条 |
| 医疗损害 | 医方证明无过错（特定情形） | 民法典第1222条 |

#### 判断流程

```
1. 先确定案由（买卖合同？产品责任侵权？）
2. 再确定举证责任分配（一般规则？倒置？）
3. 再确定我方抗辩方向（否认？过错相抵？）
4. 最后决定是否主动申请鉴定
```

**示例**：
- 买卖合同质量纠纷 → 一般规则 → 原告申请鉴定
- 产品责任侵权 → 举证责任倒置 → 被告（生产者）必须申请鉴定证明无缺陷
- 买卖合同但被告主张过错相抵 → 被告需自行举证原告过错

**教训**：不能把规则绝对化，要具体问题具体分析。用户原话："被告不是决不能 是要视案件案由 案件性质 争议标的而定 不是一杆子打死全都由原告提出 还有举证责任倒置的周况你要自我学习啊"

**正确做法**：
1. 否认质量缺陷存在，把举证责任推回原告
2. 质疑原告证据（是否有鉴定报告？变形原因是否明确？）
3. 如果原告申请鉴定，我方配合但对鉴定事项、鉴定机构提出意见
4. 如果鉴定结论不利，可申请重新鉴定或补充鉴定
5. 如果原告无法证明质量缺陷，请求驳回诉请

**被告的正确鉴定策略**：

| 阶段 | 被告行动 | 目的 |
|------|----------|------|
| 答辩阶段 | 主张原告举证不充分 | 迫使原告承担鉴定成本 |
| 举证阶段 | 提交被告证据 | 证明原告过错 |
| 庭审阶段 | 质疑原告证据，要求原告申请鉴定 | 举证责任在原告 |
| 鉴定阶段 | 对原告鉴定结论提出质证 | 如有异议申请重新鉴定 |

**庭审话术**：
> "审判长，我方不认可设备存在质量问题。根据谁主张谁举证的原则，原告主张设备有质量缺陷，应当由原告举证证明。原告仅提供照片和自行委托的维修记录，未能证明设备变形系制造缺陷导致，应承担举证不能的法律后果。"

**注意**：如果被告主张"原告存在过错"（过错相抵），则该主张的举证责任在被告方。

### 1.2 举证责任分配速查表

| 待证事实 | 举证责任方 | 说明 |
|----------|-----------|------|
| 合同关系存在 | 主张方 | 合同文本、签章 |
| 对方违约 | 主张方 | 违约事实+因果关系 |
| 产品质量缺陷 | **原告（买方）** | 鉴定报告、检测记录 |
| 损失金额 | 主张方 | 费用凭证、计算依据 |
| 对方存在过错（过错相抵） | 被告 | 过错事实+因果关系 |
| 已履行合同义务 | 主张方 | 履行凭证 |

## 二、法条引用错误（高发错误）

### 2.1 凭记忆引用法条

**错误表现**：
- 引用已废止的旧法条文（如旧版买卖合同司法解释第12/14/31条）
- 引用条文内容与原文不符（如民法典第592条"或者扩大"是加的）
- 引用"合同法"条文（已被民法典取代）
- 凭记忆引用，未经MCP工具核实

**正确做法**：
1. **法条引用必须用MCP工具核实原文**，不能凭记忆
2. 使用 `mcp_pkulaw_get_article(title="中华人民共和国民法典", number="第XXX条")` 获取准确条文
3. 使用 `mcp_pkulaw_search_article(text="关键词")` 搜索相关法条
4. 使用 `mcp_yuandian_search_fatiao(keyword="法规名称 条号")` 交叉验证
5. 引用时标注来源：`来源：北大法宝` 或 `来源：元典智库`

**核实流程**：
```
1. 先用 mcp_pkulaw_get_article 获取条文原文
2. 对比我方文件中的引用
3. 如有差异，立即修正
4. 在文件中标注"来源：北大法宝"和核实日期
```

## 五、把核实工作推给律师（推卸责任）

**错误表现**：
- 写"仍需人工核实"
- 写"建议余律师确认"
- 写"需当事人提供证据"
- 说"我不确定，让我确认一下"后不行动

**正确做法**：
- 有MCP工具就自己核实，不推给律师
- 北大法宝、元典智库、企查查等工具已配置，必须使用
- 不确定就说"我不确定"，然后立即用工具查证
- 需要当事人提供的证据，明确列出清单，不模糊表述

**用户原话**（2026-06-04）：
> "你给我装了这么多mcp和skill是让你调用的，你去查"
> "不要一有问题推给我"

**铁律**：有工具不用是失职，把工作推给用户是失职。

## 四、案件分析只讲理论不讲实务

**错误表现**：
- 只分析法律关系、法条适用
- 不涉及证据准备、程序要求、庭审策略
- 律师拿到分析报告后无法直接用于办案

**正确做法**：
每个案件分析必须覆盖7个维度：
1. 法律依据（法条+司法解释+时效性）
2. 裁判规则（类案+案号+启示）
3. 举证责任（谁主张谁举证）
4. 证据准备（清单+来源+获取方式）
5. 时间节点（举证期限+开庭+上诉）
6. 程序要求（管辖+回避+鉴定+保全）
7. 风险提示（最坏结果+应对预案）

## 五、绝对化思维（致命错误）

**错误表现**：
- 写"被告绝不能主动申请鉴定"
- 写"所有案件都适用xxx规则"
- 写"必须xxx，否则xxx"

**正确做法**：
- 法律规则有原则就有例外
- 要视案件案由、性质、争议标的而定
- 用"一般规则"代替"绝对规则"
- 列出例外情形

**教训**：
> 法律没有绝对，只有具体问题具体分析。把规则绝对化会误导律师，造成严重后果。

## 六、数据未核实（赔偿计算器教训）

**错误表现**：
- 赔偿标准数据用错（江苏人均可支配收入66172→68956）
- 年度标注错误（2024→2025→2026）
- 假装能联网获取权威数据，实际无法访问统计局网站
- 没有问律师确认就直接使用
- 工伤补助金用月数计算，实际是固定金额

**正确做法**：
1. 涉及赔偿标准、统计数据，必须问余律师确认
2. 无法联网获取权威数据时，直接说"我无法获取，请您提供"
3. 数据来源标注清楚："余律师确认"或"待核实"
4. 年度要准确：2026年公布 = 2025年度数据
5. 工伤补助金核实：用元典智库查询各省《实施〈工伤保险条例〉办法》原文

**铁律**：
> 数据错误会导致赔偿金额计算错误，直接影响案件结果。宁可多问一句，不能用错数据。

## 七、不主动使用工具核实（用户期望）

**用户原话**："我给你装了这么多mcp和skill是让你调用的 你去查"

**错误表现**：
- 有北大法宝、元典智库、企查查等MCP工具，却凭记忆写
- 等用户提供数据才修正，不主动核实
- 把核实工作推给律师

**正确做法**：
1. 涉及法条、案例、企业信息，**主动调用MCP工具核实**
2. 涉及赔偿标准，用元典智库查询法规原文
3. 涉及企业信息，用企查查MCP查询
4. 不确定就查，查不到再问用户

**用户期望**：
> 余律师期望AI助手是"超级大脑"，能主动学习、主动核实、主动发现错误并修正。不是被动等任务触发，而是主动使用工具。

## 八、验证不够彻底（用户期望）

**用户原话**："三个模块在仔细核查两边遍 不要担心token"

**错误表现**：
- 只做一遍验证就认为完成
- 担心token消耗而省略验证步骤
- 边界情况没有测试

**正确做法**：
1. 用户要求"仔细核查"时，至少做三遍验证
2. 第一遍：基础计算验证（核心公式）
3. 第二遍：不同参数验证（不同省份、不同等级）
4. 第三遍：边界情况验证（年龄边界、金额边界、年限边界）
5. 不要担心token消耗，用户明确说"不要担心token"

**铁律**：
> 用户要求彻底验证时，宁可多验几遍，不能漏掉问题。

## 九、工具使用陷阱

### 9.1 read_file 添加行号导致文件损坏（GitHub Pages 教训）

**错误表现**：
- 使用 `hermes_tools.read_file()` 读取文件后，内容自动带有行号（如 `1|<!DOCTYPE html>`）
- 将带行号的内容写回文件后，HTML/CSS/JS 文件无法正常渲染
- GitHub Pages 部署后显示混乱的数字+竖线

**正确做法**：
1. 使用 `read_file` 读取后，必须用正则删除行号再写回：`re.sub(r'^\s*\d+\|', '', content, flags=re.MULTILINE)`
2. 或者直接用 `terminal` 的 `cat`/`head` 命令读取，不带行号
3. **生成新文件时，直接用 `write_file` 写入干净内容，不要经过 `read_file` → 修改 → `write_file` 的流程**
4. 写入后用 `terminal` 的 `head -5` 验证文件前5行是否干净

**铁律**：
> `read_file` 的行号是显示用的，不是文件内容。永远不要把 read_file 的输出直接写回文件。

### 9.2 GitHub Pages 部署陷阱

**常见问题**：
1. `/docs` 文件夹部署：Branch 选 `master`，文件夹选 `/docs`
2. Save 按钮灰色：需要先选择文件夹（`/root` 或 `/docs`），Save 才会变亮
3. 浏览器缓存：部署后需要强制刷新（`Cmd+Shift+R`）或用无痕模式
4. 部署延迟：GitHub Pages 需要 1-2 分钟部署
