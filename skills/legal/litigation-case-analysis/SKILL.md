# 诉讼案件分析技能

面向中国诉讼律师的案件分析框架，集成元典智库和北大法宝 MCP 工具。

## 适用场景

- 新案接案分析
- 请求权基础确定
- 证据组织与验证
- 法律检索与案例研究
- 知识卡片生成与沉淀

## MCP 工具

### 企查查 QCC（6个Server，约180个工具）

企业尽调专用工具集，覆盖工商、税务、司法、知识产权、招投标全维度。
详见 `references/qcc-enterprise-due-diligence.md`。

**诉讼场景速查**：
- 对手画像 → `get_company_registration_info` + `get_judicial_documents` + `get_dishonest_info`
- 执行评估 → `get_judgment_debtor_info` + `get_high_consumption_restriction` + `get_equity_freeze`
- 关键人物背调 → `mcp_qcc_executive_*` 系列（需双参数：企业名+人名）

**报告模板**：`templates/enterprise-due-diligence-report.md`

### 元典智库（5个工具）
- `mcp_yuandian_search_fagui` — 法规检索
- `mcp_yuandian_search_fatiao` — 法条检索
- `mcp_yuandian_search_qwal` — 案例检索
- `mcp_yuandian_get_fagui_detail` — 法规详情
- `mcp_yuandian_get_fatiao_detail` — 法条详情

### 北大法宝（9个 HTTP 服务）
- `mcp_pkulaw_case_get_case_list` — 案例列表查询
- `mcp_pkulaw_case_search_search_case` — 案例智能检索
- `mcp_pkulaw_case_number_recognition_anhao_recognition` — 案号识别
- `mcp_pkulaw_law_get_law_list` — 法规列表查询
- `mcp_pkulaw_law_keyword_get_law_list` — 法规关键词检索
- `mcp_pkulaw_law_recognition_law_recognition` — 法规识别
- `mcp_pkulaw_search_article` — 法条语义检索
- `mcp_pkulaw_get_article` — 法条精确获取
- `mcp_pkulaw_doc_link_get_linked_content` — 文档链接

## 执行步骤

### 步骤零：对手企业尽调（诉讼案件必做）

对诉讼对手进行企业尽调，获取工商、司法、资质、知识产权全维度数据。
详见 `references/qcc-enterprise-due-diligence.md`，报告模板 `templates/enterprise-due-diligence-report.md`。

沉淀路径：`法律知识库/01-案件笔记/{企业名称}-企业尽调报告.md`

### 步骤一：案件基本信息采集

收集以下信息：
- 当事人（原告、被告、第三人）
- 案由
- 诉讼请求
- 事实与理由
- 管辖法院
- 标的金额

### 步骤二：请求权基础分析

1. **识别法律关系性质**
   - 合同纠纷？侵权纠纷？物权纠纷？公司纠纷？
   - 具体案由是什么？

2. **确定请求权基础**
   - 谁向谁主张？（主体）
   - 依据什么主张？（法律依据）
   - 主张什么？（诉讼请求）
   - 请求权基础是什么？（民法典哪一条）

3. **诉讼策略选择**
   - 诉讼 vs 仲裁 vs 调解
   - 诉讼请求是否合理
   - 是否有反诉可能

### 步骤三：证据组织与四性验证

1. **证据收集**
   - 合同/协议
   - 付款凭证
   - 沟通记录（微信、邮件、短信）
   - 证人证言
   - 鉴定报告
   - 其他书证、物证、视听资料

2. **四性验证**
   - **真实性**：是否原件？是否有篡改？
   - **合法性**：取证方式是否合法？
   - **关联性**：与待证事实是否相关？
   - **充分性**：能否形成完整证据链？

3. **证据清单编制**
   | 序号 | 证据名称 | 证据来源 | 证明目的 | 页码 |
   |------|----------|----------|----------|------|

### 步骤四：法律检索（三轮递进）

**第一轮：双源交叉检索**

使用元典智库：
```
mcp_yuandian_search_fagui(keyword="相关法规关键词")
mcp_yuandian_search_fatiao(keyword="法规名称 条号")
mcp_yuandian_search_qwal(keyword="案由 争议焦点 地域 年份")
```

使用北大法宝：
```
mcp_pkulaw_law_get_law_list(title="相关法规关键词")
mcp_pkulaw_case_search_search_case(text="案由 争议焦点")
```

**第二轮：裁判规则验证**

对检索到的案例，提取裁判要旨，验证是否支持我方主张。

**第三轮：类案补充**

检索同一法院、同一法官的历史裁判，分析裁判倾向。

### 步骤五：知识卡片生成

将案件分析结果生成结构化知识卡片：

```markdown
# 案件知识卡片

## 基本信息
- 案号：（202X）XXX民初XXX号
- 案由：XXX纠纷
- 当事人：原告XXX vs 被告XXX
- 管辖法院：XXX人民法院

## 请求权基础
- 请求权主体：原告XXX
- 请求权客体：被告XXX
- 请求权基础：《民法典》第XXX条
- 诉讼请求：1. XXX 2. XXX

## 争议焦点
1. XXX
2. XXX

## 裁判规则
- 规则1：XXX（来源：XXX案例）
- 规则2：XXX（来源：XXX法规）

## 证据清单
| 序号 | 证据名称 | 证明目的 |
|------|----------|----------|
| 1 | XXX | XXX |

## 办案经验
- 注意事项：XXX
- 风险点：XXX
- 策略建议：XXX
```

### 步骤六：多维度分析框架（强制覆盖）

每个案件分析必须覆盖以下10个维度，**实务操作层面不能省略**：

| 维度 | 内容 | 实务要点（必须写清楚） |
|------|------|----------------------|
| **① 法律依据** | 法条、司法解释、会议纪要 | 全称+条号+原文，标注时效性 |
| **② 裁判规则** | 类案检索、裁判要旨 | 案号+法院+裁判日期+对本案的启示 |
| **③ 举证责任** | 谁主张谁举证、举证责任倒置 | 明确每一项事实由谁举证，不能搞反 |
| **④ 证据准备** | 我方需准备什么、对方可能提交什么 | 证据清单+证据来源+获取方式 |
| **⑤ 时间节点** | 举证期限、开庭时间、上诉期限 | 期限台账，设置提醒，不能错过 |
| **⑥ 程序要求** | 管辖、回避、鉴定、保全 | 程序瑕疵可能导致败诉 |
| **⑦ 风险提示** | 最坏结果、应对预案 | 坦诚告知，不回避不利因素 |
| **⑧ 对手分析** | 对方诉讼画像、诉讼风格、和解动机 | 企业尽调+历史诉讼分析（必做步骤零） |
| **⑨ 调解策略** | 和解方案、谈判筹码、底线预期 | 经济分析+方案对比+谈判话术 |
| **⑩ 执行评估** | 对方偿债能力、执行风险、财产线索 | 企查查风险维度扫描（失信/被执行/限高/冻结） |

**输出要求**：
- 不能只写"法律依据"和"裁判规则"就结束
- 每个维度必须有具体内容，不能写"略"或"待补充"
- 特别是特别是③④⑥⑦⑧⑨⑩必须写清楚"怎么做"
- 维度维度⑧⑨⑩覆盖"打不打得赢"→"要不要和解"→"赢了拿不拿得到钱"的完整链条

**参考文件**：
- `references/burden-of-proof-rules.md` — 举证责任规则与实务策略
- `references/multi-dimensional-analysis.md` — 10维度分析框架详解

## 质量标准

### 四性验证
1. **准确性** — 法条条文、案号、法院名称准确无误
2. **真实性** — 引用的法规、案例真实存在
3. **合法性** — 法律依据符合现行有效法律规定
4. **关联性** — 引用内容与待解决问题有实质关联

### 引用规范
- 法条：全称+条号+内容
- 案例：案号+法院+裁判要旨
- 来源：标注检索平台和检索时间

## 安全要求

- 当事人敏感信息禁止输出到外部请求
- 法律检索前先脱敏处理
- 交付前四性验证

## 常见错误（Pitfalls）

### 1. MCP 工具名称错误
**错误**：假设北大法宝工具名为 `mcp_pkulaw_search_law` 等
**正确**：北大法宝是 9 个独立的 HTTP 服务，工具名由 MCP 客户端动态发现，格式为 `mcp_<server_name>_<tool_name>`
**验证方法**：调用工具前先确认工具是否在 `hermes mcp list` 中

### 2. 敏感信息泄漏
**错误**：在技能文件中写入 API Key、Token、当事人信息
**正确**：技能文件中只引用 MCP 工具名，不包含任何凭证
**示例**：
- ❌ `mcp_yuandian_search_fagui(keyword="民法典", api_key="sk_lBO...nSFo")`
- ✅ `mcp_yuandian_search_fagui(keyword="民法典")`

### 3. 法律检索结果未验证
**错误**：直接引用检索结果作为法律依据
**正确**：必须四性验证（准确性、真实性、合法性、关联性）
**验证步骤**：
1. 确认法规时效性（是否现行有效）
2. 确认案例有效性（是否被撤销或改判）
3. 交叉验证（元典+北大法宝双源）
4. 标注来源和检索时间

### 4. 举证责任分配错误 — 特别是鉴定申请权（极高频！）
**错误表现**：
- 被告主动申请司法鉴定证明设备无质量问题
- 在诉讼策略中写"我方申请鉴定"、"鉴定施压"、"申请多项鉴定"
- 在调解谈判中说"我们宁愿申请鉴定"

**正确原则**：根据"谁主张谁举证"原则（民事诉讼法第67条），原告主张设备有质量缺陷，应由原告举证证明，包括申请鉴定。但要视案件案由、性质而定，不能一杆子打死。

**被告应主动申请鉴定的情形**：
1. 被告主张过错相抵（需申请鉴定证明原告过错）
2. 被告主张设备符合标准（主动申请可选择有利鉴定机构）
3. 预判原告会申请（不如被告先申请，掌握主动权）
4. 争议标的较大（鉴定结论对案件走向影响大时）

**举证责任倒置的例外**：产品责任侵权、环境污染侵权、高度危险作业、医疗损害等适用举证责任倒置。

**正确做法**：
1. 否认质量缺陷存在，把举证责任推回原告
2. 质疑原告证据（是否有鉴定报告？变形原因是否明确？）
3. 如果原告申请鉴定，我方配合但对鉴定事项、鉴定机构提出意见
4. 如果鉴定结论不利，可申请重新鉴定或补充鉴定
5. 如果原告无法证明质量缺陷，请求驳回诉请

**被告的正确鉴定策略**（要视案件情况而定）：

| 情形 | 被告行动 | 说明 |
|------|----------|------|
| 原告主张质量问题（一般规则） | 不主动申请，质疑原告举证 | 举证责任在原告 |
| 被告主张过错相抵 | 需自行申请鉴定 | 举证责任在被告 |
| 产品责任侵权（举证责任倒置） | 被告需证明无缺陷 | 法定例外 |
| 预判原告会申请 | 可主动申请，掌握主动权 | 策略选择 |

**庭审话术**：
> "审判长，我方不认可设备存在质量问题。根据谁主张谁举证的原则，原告主张设备有质量缺陷，应当由原告举证证明。原告仅提供照片和自行委托的维修记录，未能证明设备变形系制造缺陷导致，应承担举证不能的法律后果。"

**注意**：如果被告主张"原告存在过错"（过错相抵），则该主张的举证责任在被告方。

### 5. 企查查多候选实体未等用户确认
**错误**：`get_company_by_query` 返回多个候选时自动选择第一个
**正确**：必须将候选列表完整展示给用户，等待用户明确指定后再调用下游工具
**后果**：自动选择可能导致查询错误主体，产生严重尽调风险
**例外**：唯一精确匹配时可直接使用

### 6. 企查查高管查询漏传 personName
**错误**：用 `mcp_qcc_executive_*` 系列工具时只传 searchKey
**正确**：`mcp_qcc_executive_*` 需要双参数锚定：`searchKey`（企业名/信用代码）+ `personName`（董监高姓名）
**后果**：漏传 personName 会导致查询失败或返回无关数据

### 7. 法条引用未核实（极高频！）
**错误表现**：
- 引用已废止的旧法条文（如旧版买卖合同司法解释第12/14/31条）
- 引用条文内容与原文不符
- 引用"合同法"条文（已被民法典取代）
- 凭记忆引用，未经MCP工具核实

**正确做法**：
1. **法条引用必须用MCP工具核实原文**，不能凭记忆
2. 使用 `mcp_pkulaw_get_article(title="中华人民共和国民法典", number="第XXX条")` 获取准确条文
3. 使用 `mcp_pkulaw_search_article(text="关键词")` 搜索相关法条
4. 使用 `mcp_yuandian_search_fatiao(keyword="法规名称 条号")` 交叉验证
5. 引用时标注来源：`来源：北大法宝` 或 `来源：元典智库`

**特别注意**：
- **买卖合同司法解释**：2020年修正版删除了旧版第2/3/4/15/16/18/28/30/32/35/37/41/43条，条文顺序重新调整。旧版第12/14/31条等已不存在，不能引用。
- **合同法**：已被民法典取代，引用时应使用民法典条文
- **法条原文差异**：如民法典第592条，原文是"对方对损失的发生有过错的"，不是"对损失的发生或者扩大也有过错的"

**核实流程**：
```
1. 先用 mcp_pkulaw_get_article 获取条文原文
2. 对比我方文件中的引用
3. 如有差异，立即修正
4. 在文件中标注"来源：北大法宝"和核实日期
```

### 8. 案件分析只讲理论不讲实务
**错误**：只分析法律关系、法条适用，不涉及证据准备、程序要求、庭审策略
**正确**：每个案件分析必须覆盖7个维度（见"多维度分析框架"），特别是实务操作层面不能省略
**后果**：律师拿到分析报告后无法直接用于办案，还需要自己补充大量实务细节

## ⚠️ 血泪教训

详见 [[legal-analysis-pitfalls]] 技能，记录了法律分析中极易犯的错误和教训。

**核心教训**：
1. 举证责任分配要视案件案由、性质而定，不能一杆子打死（买卖合同vs产品责任侵权vs举证责任倒置）
2. 法条引用必须用MCP工具核实原文，不能凭记忆（旧法/合同法/条文内容错误）
3. 不能把核实工作推给律师，有工具就自己用
4. 涉及赔偿标准数据，必须用MCP工具核实，不能凭记忆写错误数据
